# 前端编码规范

本文档是针对 前端的 HTML、JavaScript、CSS 设计的，但是在使用各种 JavaScript 的预编译语言时(如 TypeScript 等)时，适用的部分也应尽量遵循本文档的约定。

# 注释

注释就是对代码的解释和说明，其目的是让人们能够更加轻松地了解代码。

注释是编写程序时，写程序的人给一个语句、程序段、函数等的解释或提示，能提高程序代码的可读性。

注释只是为了提高可读性，不会被计算机编译。

# 1.1 单行注释

[强制] 必须独占一行，// 后跟一个空格，缩进与下一行被注释说明的代码一致。

  • 使用方法
  // describe...
1

# 1.2 文件注释

[强制] 文件顶部必须包含文件注释，用 @file 标识文件说明。

  • 使用方法
  /**
  * @file Describe the file
  **/
1
2
3

[建议] 文件注释中可以用 @author 标识开发者信息。

开发者信息能够体现开发人员对文件的贡献，并且能够让遇到问题或希望了解相关信息的人找到维护人。通常情况文件在被创建时标识的是创建者。随着项目的进展，越来越多的人加入，参与这个文件的开发，新的作者应该被加入 @author 标识。

@author 标识具有多人时，原则是按照 责任 进行排序。通常的说就是如果有问题，就是找第一个人应该比找第二个人有效。比如文件的创建者由于各种原因，模块移交给了其他人或其他团队，后来因为新增需求，其他人在新增代码时，添加 @author 标识应该把自己的名字添加在创建人的前面。

@author 中的名字不允许被删除。任何劳动成果都应该被尊重。

业务项目中，一个文件可能被多人频繁修改，并且每个人的维护时间都可能不会很长，不建议为文件增加 @author 标识。

对于业务逻辑无关的技术型基础项目，特别是开源的公共项目，应使用 @author 标识。

  • 使用方法
  /**
  * @file Describe the file
  * @author author-name(mail-name@domain.com)
  *         author-name2(mail-name2@domain.com)
  **/

# 1.3 函数描述

[强制] 函数/方法注释必须包含函数说明，有参数和返回值时必须使用注释标识。

i. 当 return 关键字仅作退出函数/方法使用时，无须对返回值作注释标识。

ii. [强制] 参数和返回值注释必须包含类型信息，且不允许省略参数的说明。

iii. [建议] 当函数是内部函数，外部不可访问时，可以使用 @inner 标识。

  • 使用方法
  /**
  * 函数描述
  *
  * @param {string} p1 参数1的说明
  * @param {string} p2 参数2的说明，比较长
  * 那就换行了.
  * @param {number=} p3 参数3的说明（可选）
  * @return {Object} 返回值描述
  **/
function foo (p1, p2, p3) {
    var p3 = p3 || 10;
    return {
        p1: p1,
        p2: p2,
        p3: p3
    };
}

# 1.4 细节注释

对于内部实现、不容易理解的逻辑说明、摘要信息等，我们可能需要编写细节注释。

[建议] 细节注释遵循单行注释的格式。说明必须换行时，每行是一个单行注释的起始。

  • 使用方法
  function foo(p1, p2, opt_p3) {
    // 这里对具体内部逻辑进行说明
    // 说明太长需要换行
    for (...) {
      ....
    }
  }

# 1.5 Vue 中 data() 中的变量注释

在实际使用 Vue 构建项目的时候，data 中的每个变量需要给予变量说明，方便后期代码维护。

[建议] Vue 中 data() 中的变量注释遵循单行注释的格式。说明必须换行时，每行是一个单行注释的起始。

  • 使用方法
  data () {
    return {
      // 高级搜索
      searchData: {
        drugName: '',         // 商品名称
        approvalNo: '',       // 批准文号
        drugCode: '',         // 商品码
        manufacturer: '',     // 生产厂家
        supervisionCode: '',  // 监管码
        insuranceCode: ''     // 国家医保
      }
    }
  }

# 命名规则

应反映该元素的功能或使用通用名称，而不要用抽象的晦涩的命名（原则：见名知其义）

# 2.1 文件命名规则

注：文件名不要以中文或人名命名，可以在文件内部注释说明是谁修改或建立的这个文件。

文件夹采用 “大驼峰命名法”

  • 使用方法
  例：DataBaseUser：变量 DataBaseUser 单词首字母大写。
1

# 2.2 id 命名规则

id 采用 “小驼峰命名法”，变量第一个单词是全部小写，后面的单词首字母大写。

  • 使用方法
  <section id="nav">
    <div id="navTitle">
      ...
    </div>
  </section>

# 2.3 class 命名规则

class 采用 “中划线法命名法，class 类名中的每一个逻辑断点都有一个中划线来标记。

  • 使用方法
  <section class="nav">
    <div class="nav-title">
      ...
    </div>
  </section>

# 2.4 class、id 命名注意事项

  • 命名不允许使用拼音首字母缩写；
  • 不允许通过1、2、3 等序号进行命名；
  • 避免 class 与 id 重名；
  • id 注意用于标识模块或页面的某一个父容器区域，名称必须唯一，不要随意新建id；
  • class 用于标识某一个类型的对象，命名必须言简意赅；
  • 尽可能提高代码模块的复用，样式尽量用组合的方式；
  • 规则名称中不应该包含颜色、定位等与具体显示效果相关的信息，应该用意义命名，而不是结果名称。

# 2.5 函数方法命名规范

  • 命名方法： 小驼峰式命名法
  • 命名规范： 前缀应该为动词
  • 命名建议： 常用动词约定

JavaScrip函数名的命名规范严格遵循以下5条：

  • 首字符必须是字母、下划线、，后跟任意的字母、数字、下划线、；
  • 严格区分大小写；
  • 不能使用系统的关键字和保留字；
  • 命名尽量有意义；
  • 推荐使用驼峰命名法。

动词含义

动词 说明 实例
can 判断是否可以执行某个动作 canSomeEvent
get 获取某个值 getSomeData
handle 处理某个需要执行的事件，方法 handleSomeEvent
has 判断是否存在或者包含某个值 hasSomeData
is 判断是否存在或者为某个值 isSomeData
load 加载某些数据 loadSomeData
set 设置某个值 setSomeData

# Vue 项目书写规范

关于 Vue 生命周期方法放置顺序在项目中的书写顺序，按照下面的内容统一进行规范。

HTML 部分包含在 template 中，每个 Vue 文件需要添加文件注释

  *         author-name2(mail-name2@domain.com)
  **/
